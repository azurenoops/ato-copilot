## Requirement Completeness

- [x] CHK001 — Are requirements defined for what happens when a PTA is re-run on a system that already has a PTA? (overwrite vs. version?) [Completeness, Gap, Spec §Cap 1.1] — **Resolved**: Overwrite behavior. Data-model enforces one-PTA-per-system unique constraint. T019 `InvalidatePtaAsync` deletes existing PTA. Clarification Q6 confirms auto-invalidation on info-type changes.
- [x] CHK002 — Are requirements specified for PIA section-level completion tracking (e.g., can a PIA be submitted for review with unanswered sections)? [Completeness, Gap, Spec §Cap 1.2] — **Resolved**: Incomplete sections permitted. PIA `Sections[].Answer` is nullable. ISSM reviews assess content quality, not section completeness — consistent with OMB M-03-22 narrative format. No completeness gate on review submission.
- [x] CHK003 — Are requirements defined for PIA version management beyond the `Version` field? (e.g., can previous versions be retrieved? are diffs tracked between revisions?) [Completeness, Gap, Spec §Cap 1.4] — **Resolved**: Version field tracks revision count only. Previous versions are not retained — PIA is a living document overwritten on each revision. Historical tracking is out of scope per Part 9 ("Not a document storage system").
- [x] CHK004 — Are mobile/responsive requirements documented for the privacy compliance dashboard, or is it MCP-only? [Completeness, Spec §Cap 4.5] — **Resolved**: MCP-only. Dashboard is an MCP tool (`compliance_check_privacy_compliance`) returning JSON. No UI rendering responsibility — consistent with the MCP server architecture and Part 2 Interfaces table.
- [x] CHK005 — Are requirements specified for what data populates the `AffectedIndividuals` and `EstimatedRecordCount` fields in auto-detection mode? [Completeness, Gap, Data Model §PTA] — **Resolved**: These fields are populated only in manual mode. Auto-detection populates `PiiCategories` and `PiiSourceInfoTypes` from SecurityCategorization. Both fields are nullable (`string?`, `int?`) — optional for auto-detection.
- [x] CHK006 — Are requirements defined for interconnection deletion (vs. termination)? Can a mistakenly-created interconnection be removed entirely, or only terminated? [Completeness, Gap, Spec §Cap 2.4] — **Resolved**: Deletion intentionally not supported. Only termination with audit trail retention. Consistent with DoDI 8510.01 record retention requirements and Part 9 exclusions.
- [x] CHK007 — Are requirements specified for agreement renewal workflow? (e.g., create new agreement vs. extend expiration on existing) [Completeness, Gap, Spec §Cap 3.2] — **Resolved**: Renewal via either creating a new agreement (`compliance_register_agreement`) or updating expiration on existing agreement (`compliance_update_agreement`). Cap 3.5 supports full lifecycle status transitions.
- [x] CHK008 — Are requirements defined for the `compliance_update_interconnection` tool's behavior when updating a terminated interconnection? (reactivation possible?) [Completeness, Gap, Contracts §compliance_update_interconnection] — **Resolved**: Terminated interconnections can have non-status fields corrected but cannot be reactivated (one-way terminal status). Contract allows field updates; Cap 2.4 states terminated records are "retained for audit trail but excluded from active counts."

## Requirement Clarity

- [x] CHK009 — Is "pre-populated answers where data exists" in Cap 1.2 quantified — which specific fields are pre-populated and from which sources? [Clarity, Spec §Cap 1.2] — **Resolved**: Three explicit sources: system description (`RegisteredSystem.Description`), PII categories (`PTA.PiiCategories`), safeguards (control baseline). PiaSection entity tracks origin via `IsPrePopulated` + `SourceField`. Contract example shows `sourceField` values per section.
- [x] CHK010 — Is "AI-assisted narrative drafting" in Cap 1.3 specified with prompt strategy, token limits, or quality expectations? [Clarity, Spec §Cap 1.3] — **Resolved**: Section-by-section prompt strategy (research.md R2, 8 OMB M-03-22 sections). Template-only fallback when AI unavailable (plan.md Risk Assessment). Performance SLA: PIA generation < 30s. Token limits and prompt engineering are implementation details, not spec requirements.
- [x] CHK011 — Is the 90-day expiration warning threshold in Cap 3.3 configurable, or is it fixed? Is this documented as a requirement? [Clarity, Spec §Cap 3.3] — **Resolved**: Fixed at 90 days. Aligns with DoDI 8510.01 standard renewal notification period. Configurability is an enhancement, not MVP scope.
- [x] CHK012 — Is "validate against authorization boundary — the target must be external" in Cap 2.1 specified with a concrete validation algorithm? (How is "external" determined from `AuthorizationBoundary` records?) [Clarity, Spec §Cap 2.1] — **Resolved**: User-assertion based. Interconnections are manually registered (Part 9: "not a network topology mapper"). The system trusts user input about target externality — no automated boundary cross-reference. Consistent with manual documentation approach.
- [x] CHK013 — Is "annual review reminders" in Cap 1.4 specified with delivery mechanism, recipient, and timing? (email? dashboard alert? how many days before expiration?) [Clarity, Spec §Cap 1.4] — **Resolved**: Delivery via ConMon monitoring cycle (Cap 3.4). PIA expiration detected automatically during ConMon checks. Expired PIAs → status set to Expired + `SignificantChange` record created (T052). No email — ConMon reports and dashboard surface the status.
- [x] CHK014 — Is the "advisory warnings" behavior for legacy systems (already past Prepare) defined with specific output format and visibility? [Clarity, Spec §Cap 4.1] — **Resolved**: Advisory warnings surfaced via `compliance_check_privacy_compliance` dashboard tool. Returns `overallStatus` = "ActionRequired" with individual gate satisfaction flags. No blocking action — informational JSON response only.

## Requirement Consistency

- [x] CHK015 — Are persona-to-RBAC mappings consistent between Part 4 (Personas & Needs) and the RBAC Summary Matrix in contracts? (e.g., Part 4 says ISSO conducts PTAs, but RBAC shows Analyst/SecurityLead — is ISSO mapped to Analyst?) [Consistency, Spec §Part 4 vs. Contracts §RBAC Matrix] — **Resolved**: Follows existing codebase RBAC convention: ISSO→Analyst, ISSM→SecurityLead, SCA→Auditor, AO→AuthorizingOfficial, Engineer→PlatformEngineer. Explicit mapping table added per CHK043.
- [x] CHK016 — Is the interconnection initial status consistent? Cap 2.1 says "Active status" for new interconnections, but the contract returns `"status": "Proposed"`. Which is correct? [Consistency, Spec §Cap 2.1 vs. Contracts §compliance_add_interconnection] — **Fixed**: Contract is authoritative — new interconnections start as `Proposed` (pre-authorization). AT-06 corrected from "Active status" to "Proposed status" in spec.md. Proposed→Active transition requires explicit `compliance_update_interconnection` call.
- [x] CHK017 — Is PIA status lifecycle consistent across all references? Cap 1.4 says "Draft → UnderReview → Approved → Expired" but Cap 1.5 and review flow show direct Draft → Approved (skipping UnderReview). When does `UnderReview` status get set? [Consistency, Spec §Cap 1.4 vs. §Cap 1.5] — **Resolved**: Consistent. `UnderReview` is set by `InvalidatePtaAsync` when an Approved PIA needs re-analysis due to info-type changes (Clarification Q6). Review tool accepts both Draft and UnderReview status. Draft→Approved is the normal path; UnderReview→Approved is the re-validation path.
- [x] CHK018 — Are the NIST 800-18 section numbers used consistently? Part 1 references "§10 (Interconnections)" but NIST 800-18 Rev 1 §3.2 lists system interconnections as a subsection of §3, not §10. Is the section numbering convention documented? [Consistency, Spec §Part 1 vs. §Part 3] — **Resolved**: "§10" is the organization's SSP template section numbering convention, not NIST's subsection numbering. SspService uses organizational section numbers mapped to NIST content areas. Convention is implicit in the existing codebase.
- [x] CHK019 — Is the `SignificantChange.ChangeType` value for ISA expiration consistent across spec and ConMon integration? Spec says `"New Interconnection"` but an ISA expiration is not a "new" interconnection — is this the intended reuse of an existing enum value? [Consistency, Spec §Clarifications Q5 vs. §Cap 3.4] — **Resolved**: Intentional reuse of existing ChangeType enum value. "New Interconnection" triggers reauthorization review in existing ConMon workflow. Adding a new enum value would require modifying ConMon infrastructure beyond this feature's scope. Confirmed in Clarification Q5.

## Acceptance Criteria Quality

- [x] CHK020 — Are acceptance test expected results measurable for AT-03? "PIA created in Draft status with pre-populated sections" — how many sections must be pre-populated to pass? [Measurability, Spec §AT-03] — **Resolved**: Measurable via contract — `totalSections: 8` (OMB M-03-22 per research.md R2), `prePopulatedSections` count returned. AT-03 verifies Draft status + 8 sections present + at least some pre-populated.
- [x] CHK021 — Are acceptance test expected results measurable for AT-15? "Comprehensive report" — what specific fields/metrics must appear in the SCA report? [Measurability, Spec §AT-15] — **Resolved**: Measurable via `compliance_check_privacy_compliance` contract — returns 12 specific fields: `ptaDetermination`, `piaStatus`, `privacyGateSatisfied`, `activeInterconnections`, `interconnectionsWithAgreements`, `expiredAgreements`, `expiringWithin90Days`, `interconnectionGateSatisfied`, `hasNoExternalInterconnections`, `overallStatus`, `systemId`, `systemName`.
- [x] CHK022 — Is there an acceptance test for the `PendingConfirmation` determination path? (PTA with ambiguous info types → PendingConfirmation → user confirms → determination updated → gate re-evaluated) [Coverage, Gap, Spec §Part 10] — **Fixed**: Added AT-21 — PendingConfirmation → user confirms PII → determination updated to PiaRequired. Gate blocked until PIA completed.
- [x] CHK023 — Is there an acceptance test for PTA invalidation when info types change? (Approved PIA → info types updated → PTA invalidated → PIA set to UnderReview) [Coverage, Gap, Spec §Part 10] — **Fixed**: Added AT-22 — Info types changed → PTA deleted → PIA status → UnderReview → privacy gate re-opens.
- [x] CHK024 — Is there an acceptance test for the `HasNoExternalInterconnections` certification path? (System certified with no interconnections → interconnection added → certification flag cleared) [Coverage, Gap, Spec §Part 10] — **Fixed**: Added AT-23 — System certified → interconnection added → `HasNoExternalInterconnections` cleared → Gate 4 requires agreement validation.
- [x] CHK025 — Are acceptance criteria defined for AI-generated content quality? (ISA template correctness, PIA narrative completeness — are there minimum content expectations?) [Measurability, Gap] — **Resolved**: AI generates drafts, not final artifacts. Quality gates are human review steps (ISSM reviews PIA via `compliance_review_pia`; ISA is reviewed before signing). Automated quality metrics are not required — the review workflow IS the quality gate.

## Scenario Coverage

- [x] CHK026 — Are requirements defined for concurrent PTA re-analysis while a PIA review is in progress? (ISSO re-runs PTA while ISSM is reviewing PIA — what happens to the in-flight review?) [Coverage, Exception Flow, Gap] — **Resolved**: Sequential by design. `InvalidatePtaAsync` sets PIA to UnderReview, canceling any in-flight review. Correct behavior — if underlying PII profile changes, PIA must be re-assessed. EF Core single-writer pattern prevents data corruption.
- [x] CHK027 — Are requirements defined for what happens when an interconnection's data classification is upgraded (e.g., Unclassified → Secret)? Does this require new ISA or invalidate existing? [Coverage, Alternate Flow, Spec §Cap 2.3] — **Resolved**: Classification upgrade triggers advisory warning ("Data classification changed — ISA review recommended") via `compliance_update_interconnection` contract. Existing ISA remains valid until explicitly updated or expired. No automatic invalidation — human judgment required.
- [x] CHK028 — Are requirements specified for bulk interconnection operations? (e.g., system has 20 interconnections — can agreements be validated individually or only as a batch?) [Coverage, Spec §Cap 3.3] — **Resolved**: Validation is batch-level — `compliance_validate_agreements` evaluates all active interconnections in one call. CRUD is per-interconnection. Bulk add/update not needed for typical scale (0-20 interconnections per system per plan.md Scale/Scope).
- [x] CHK029 — Are requirements defined for the PIA generation path when the system has no control baseline? (Can AI drafting proceed without safeguards data from controls?) [Coverage, Edge Case, Spec §Cap 1.3] — **Resolved**: Graceful degradation. Pre-population uses available data only — safeguards section left for manual completion if no control baseline. `PiaSection.IsPrePopulated = false`, `Answer = null`. AI drafting proceeds using other available context (system description, PII info types).
- [x] CHK030 — Are requirements specified for the `ExemptionRationale` field — is it validated against known exemption categories, or is it free-form text? [Coverage, Spec §Cap 1.1] — **Resolved**: Free-form text (MaxLength 2000). Exemption rationale varies too widely across agencies to enumerate categories. Value recorded for audit trail and ISSM review. Examples in spec: "government-to-government", "national security."

## Edge Case Coverage

- [x] CHK031 — Is the fallback behavior specified when Azure OpenAI is unavailable during PIA generation or ISA template drafting? [Edge Case, Gap, Spec §Cap 1.3] — **Resolved**: Template-only fallback documented in plan.md Risk Assessment, research.md R2, and T017. PIA/ISA generates with structured template sections but no AI-drafted narratives. Service returns valid document with empty narrative fields for manual completion.
- [x] CHK032 — Are requirements defined for a system with zero information types attempting PTA auto-detection? (Error contracts specify this case, but is the user guidance complete?) [Edge Case, Spec §Cap 1.1] — **Resolved**: Contract error case returns actionable message: "System has no information types defined. Add information types to security categorization first." Manual mode (`manual_mode = true`) available as alternative path. Error guidance is complete.
- [x] CHK033 — Are requirements defined for what happens when all agreements for an interconnection are in `Terminated` status but the interconnection itself is `Active`? [Edge Case, Gap, Spec §Cap 3.3] — **Resolved**: Treated as "Missing" agreement. `compliance_validate_agreements` checks for ≥1 Signed + non-expired agreement. All Terminated = none qualify → `validationStatus: "Missing"`. Gate 4 fails until new agreement registered.
- [x] CHK034 — Is the behavior specified when a PIA `ExpirationDate` is null? (Approved PIA without expiration — can this state exist?) [Edge Case, Gap, Data Model §PIA] — **Resolved**: ExpirationDate always set on approval (T018: `ExpirationDate = now + 1 year`). Null state only exists for Draft/UnderReview PIAs (pre-approval). ConMon safely ignores null ExpirationDate — no erroneous expiration alerts.
- [x] CHK035 — Are requirements defined for the maximum number of interconnections per system? Is there a practical limit, or is it unbounded? [Edge Case, Gap] — **Resolved**: No hard limit enforced. Plan.md Scale/Scope documents typical ceiling of ~20 per system. Performance ensured via composite index on `(RegisteredSystemId, Status)`. Pagination on list operations per constitution. Unbounded but performant.

- [x] CHK037 — Are audit trail requirements documented for PTA/PIA status changes? - [x] CHK038 — Are data retention requirements defined for terminated interconnections and expired PIAs? (How long are they retained? Is purging defined?) [NFR, Gap] — **Resolved**: Terminated records retained indefinitely for audit compliance (Cap 2.4: "retained for audit trail"). - [x] CHK039 — Are accessibility requirements defined for privacy dashboard output and PIA document format? [NFR, Gap] — **Resolved**: Not applicable for MCP server. Output is structured JSON via MCP protocol. Accessibility is a consuming-UI concern (VS Code extension, Teams Bot) — not the server-side API. PIA markdown output uses semantic heading structure for downstream rendering.
