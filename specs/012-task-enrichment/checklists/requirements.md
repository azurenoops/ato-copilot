## Requirement Completeness

- [x] CHK001 — Are requirements defined for what happens when `CreateBoardFromAssessmentAsync` is called with an assessment containing zero Open/InProgress findings? [Completeness, Gap] — ✅ Zero findings → zero tasks → zero enrichment. Existing code filters Open/InProgress and existing test `CreateBoardFromAssessmentAsync_ZeroFindings_ReturnsEmptyBoard` validates this.
- [x] CHK002 — Are requirements defined for a board where all findings have `Informational` severity, resulting in every task receiving the static fallback string instead of AI content? [Completeness, Gap] — ✅ Spec §Clarifications Q3 + data-model.md decision flow step 3 + research R9 define Informational handling with static strings.
- [x] CHK003 — Are requirements specified for tasks created manually (no `FindingId`) when lazy enrichment (US4) or on-demand tools (US2/US3) are invoked? [Completeness, Spec §US2/US4] — ✅ data-model.md decision flow step 2: "if finding == null → SKIP". Research R4 confirms: "If FindingId is null, skip enrichment."
- [x] CHK004 — Is the expected behavior specified when `kanban_generate_script` is called with an invalid `script_type` value (not in the AzureCli/PowerShell/Bicep/Terraform enum)? [Completeness, Gap] — ✅ Contract schema defines enum constraint. Codebase pattern uses `Enum.TryParse` (KanbanTools.cs L349-351) for graceful handling.
- [x] CHK005 — Are requirements documented for partial enrichment — a task that has a `RemediationScript` but null `ValidationCriteria`? Does `EnrichTaskAsync` with `force=false` regenerate only the missing field? [Completeness, Spec §Technical Design §2] — ✅ Clarified in spec §Checklist-Driven Clarifications: atomic skip by design; use `kanban_generate_validation` directly for missing validation.
- [x] CHK006 — Are requirements defined for the `RemediationScriptType` value when the NIST-template fallback is used? Should it be set to `"AzureCli"` (the default), null, or a sentinel like `"Template"`? [Completeness, Gap] — ✅ Spec §Technical Design §2 step 2: `task.RemediationScriptType = scriptType.ToString()` — uses the requested scriptType param (default AzureCli) regardless of AI/template path.
- [x] CHK007 — Is the behavior specified when `kanban_generate_validation` is called for a task that has no `RemediationScript` yet? Does it generate validation criteria without script context, or should it generate the script first? [Completeness, Gap] — ✅ `GenerateValidationCriteriaAsync` accepts optional `scriptContent` param. When null (no script exists), falls back to template. Validation is independent of script existence.
- [x] CHK008 — Are requirements defined for how `AffectedResources` (the third problem in the Problem Statement) is improved by this feature, or is that explicitly deferred? [Completeness, Spec §Problem Statement] — ✅ Implicitly deferred — not in any user story scope. Problem Statement lists it for context; this feature addresses only RemediationScript and ValidationCriteria.
- [x] CHK009 — Are logging requirements specified for each enrichment entry point (board creation, board update, on-demand tool, lazy view)? What structured log properties are required? [Completeness, Gap] — ✅ Clarified in spec §Checklist-Driven Clarifications with required structured log properties for all entry points.
- [x] CHK010 — Is there a requirement for auditing enrichment events in the task history (`TaskHistoryEntry`) when a script or validation criteria is generated or regenerated? [Completeness, Gap] — ✅ Clarified in spec: auto-enrichment does NOT add history entries (noise); on-demand US2/US3 tools MAY add entries for user-initiated regeneration.

## Requirement Clarity

- [x] CHK011 — Is "synchronous enrichment" precisely defined in terms of when `SaveChangesAsync` is called — once after all tasks are enriched, or per-task? [Clarity, Spec §Clarifications] — ✅ Spec §Technical Design §4 shows single `SaveChangesAsync` call after `EnrichBoardTasksAsync` returns. Enrichment modifies tasks in-place; one save for all.
- [x] CHK012 — Is the progress message format for board enrichment specified with enough precision to implement? "Enriching task {n}/{total}: {taskNumber} ({controlId})..." appears only in prose, not as a formal requirement. [Clarity, Spec §Technical Design §2] — ✅ Format string is sufficiently precise for implementation. Exact format is an implementation detail.
- [x] CHK013 — Is "AI vs. Template" generation method determination precisely specified? The decision flow says "based on script.IsSanitized check" but `IsSanitized` is true for both AI-generated (sanitized) and NIST-fallback scripts. [Ambiguity, Spec §Technical Design §2 / data-model.md §Decision Flow] — ✅ Clarified in spec §Checklist-Driven Clarifications: check `_aiGenerator.IsAvailable` before calling engine to determine method.
- [x] CHK014 — Is the per-task timeout of 30 seconds specified as a configurable value or hardcoded constant? Constitution Principle VIII suggests performance targets should be configurable. [Clarity, Spec §Performance Considerations] — ✅ Acceptable as hardcoded constant for dev-only tool. Research R7 establishes 30s as validated threshold.
- [x] CHK015 — Is "bounded concurrency via `SemaphoreSlim(5)`" specified as the max concurrency value, or as a configurable option? The research says "5 is a safe default" but the spec hardcodes it. [Clarity, Spec §Performance Considerations / research.md §R7] — ✅ Hardcoded constant is acceptable. Research R7 documents rationale. Can be promoted to config in future if needed.
- [x] CHK016 — Are the exact fixed strings for Informational tasks specified as constants? Both "Informational finding — no remediation required" and "STIG reference — no validation required" appear as inline string literals. [Clarity, Spec §Clarifications / §Technical Design §2] — ✅ Tasks.md T007 includes const extraction note (per F9 fix). Strings are defined in spec; implementation will extract to constants.
- [x] CHK017 — Is "AI is unavailable" precisely defined? The spec uses three variants: "`IChatClient` is null", "`AgentAIEnabled=false`", and "AI is unavailable" — are these all the same condition or distinct? [Ambiguity, Spec §US1.3 / §US2.4 / §US3.2] — ✅ Clarified in spec §Checklist-Driven Clarifications: all three refer to single condition `_aiGenerator.IsAvailable == false`.

## Acceptance Criteria Quality

- [x] CHK024 — Are US1's acceptance scenarios testable without a live Azure OpenAI endpoint? The spec mentions mock dependencies but doesn't define the mock contract for `IRemediationEngine.GenerateRemediationScriptAsync`. [Acceptance Criteria, Spec §US1] — ✅ `IRemediationEngine` is an interface testable via Moq. Testing Strategy items 1-6 explicitly mock it. Established codebase pattern.
- [x] CHK025 — US1.5 specifies "enrichment completes within 60 seconds" for 30+ findings — is this measurable given variable AI response times? Should there be a separate threshold for template-only (no AI) enrichment? [Measurability, Spec §US1.5] — ✅ Spec §Performance calculates worst case: 6 batches × 5s = 30s < 60s. Template path is near-instant. 60s is an upper bound for AI path.
- [x] CHK026 — US4.1 specifies "lazily generates and persists a remediation script" — can the persistence requirement be verified independently from the generation? What if `SaveChangesAsync` fails after generation? [Acceptance Criteria, Spec §US4.1] — ✅ Standard exception propagation. If save fails, task is enriched in-memory but not persisted. Retry-safe: lazy enrichment re-checks `RemediationScript == null` on next view.
- [x] CHK027 — US5.2 specifies "existing task scripts are not overwritten" — is this verifiable when enrichment is called on the full board? How is the "existing vs. new" determination made for board update? [Measurability, Spec §US5.2] — ✅ Spec §5 says "enrich only the newly created tasks". `EnrichTaskAsync(force=false)` skip logic protects existing tasks. Verifiable via test with pre-enriched tasks.
- [x] CHK028 — Are success/failure criteria defined for `BoardEnrichmentResult`? What constitutes a "successful" board enrichment — all tasks enriched? At least one? Zero failures? [Acceptance Criteria, Gap] — ✅ Board enrichment is best-effort. `BoardEnrichmentResult` reports `TasksEnriched/TasksFailed/TasksSkipped` counts. The operation never fails the board creation — individual task failures are isolated and logged.

## Scenario Coverage

- [x] CHK029 — Are requirements defined for concurrent lazy enrichment when multiple users call `kanban_get_task` for the same unenriched task simultaneously? Could this result in duplicate AI calls or race conditions on `SaveChangesAsync`? [Coverage, Exception Flow, Gap] — ✅ Acceptable: duplicate AI calls possible, last-write-wins on save. Low risk for a dev tool with limited concurrent users. No additional requirement needed.
- [x] CHK030 — Are requirements defined for what happens when `AgentAIEnabled` changes from `true` to `false` mid-board-enrichment (e.g., config reload)? [Coverage, Exception Flow, Gap] — ✅ Config is read at service construction (scoped DI per R5). Cannot change mid-request. No requirement needed.
- [x] CHK031 — Are requirements specified for enriching a task whose linked finding has been deleted (orphaned `FindingId`) or whose assessment was deleted after board creation? [Coverage, Exception Flow, Gap] — ✅ data-model.md decision flow step 2: "if finding == null → SKIP". DB lookup returns null for deleted finding → enrichment skipped gracefully.
- [x] CHK032 — Are requirements defined for the interaction between `kanban_remediate_task` (existing tool) and the enrichment pipeline? After enrichment populates a script, does `kanban_remediate_task` work without additional changes? [Coverage, Integration, Spec §No Changes Required] — ✅ Spec §No Changes Required: "`kanban_remediate_task` will succeed once scripts exist." Existing tool reads `task.RemediationScript` — enrichment populates it.
- [x] CHK033 — Are requirements specified for what the `kanban_get_task` response looks like during the 3–5s lazy enrichment window? - [x] CHK034 — Are requirements defined for enrichment behavior when `CreateBoardFromAssessmentAsync` is called multiple times with the same assessment (duplicate boards)? [Coverage, Alternate Flow, Gap] — ✅ Each board has its own tasks. Enrichment is per-board, per-task. Idempotent via skip logic (force=false). No cross-board interaction.
- [x] CHK035 — Are requirements specified for the `kanban_generate_script` tool when called on a task in `Done` status? Should completed tasks still allow script regeneration? [Coverage, Alternate Flow, Gap] — ✅ Intentionally unrestricted — no status gate. Analysts may want updated scripts after context changes regardless of task status.

## Ambiguities & Conflicts

- [x] CHK052 — The spec's enrichment decision flow (data-model.md) checks `task.RemediationScript != null` at step 1 but the Informational check at step 3 potentially sets it. If a task already has a script AND the finding is Informational, does step 1 skip override step 3? [Ambiguity, data-model.md §Decision Flow] — ✅ No ambiguity: step ordering is correct. [Conflict, Spec §Technical Design §4 vs. research.md §R8] — ✅ Clarified in spec §Checklist-Driven Clarifications: tools use `GetService` (optional), NOT `GetRequiredService`. Consistent with KanbanService optional injection pattern.
- [x] CHK054 — The `kanban_generate_script` error response for `SCRIPT_EXISTS` returns an error envelope, but lazy enrichment in `kanban_get_task` silently enriches without user confirmation. On-demand = explicit user action (error on conflict). Lazy = transparent background (silent). Different UX patterns by design.
