## Requirement Completeness — RMF Step 4 Coverage

- [x] CHK001 - Are all NIST SP 800-37 Rev 2 Task A-1 required SAP elements explicitly enumerated and mapped to spec sections? [Completeness, Spec §3.1] → **PASS**. §3.1 enumerates 15 SAP sections covering all Task A-1 elements: scope, methodology, team, schedule, boundary, ROE, control matrix, STIG testing plan, risk approach, evidence requirements.
- [x] CHK002 - Are DoDI 8510.01 Enclosure 6 SAP template requirements referenced, and is any deviation from the DoD template documented and justified? [Completeness, Gap] → **PASS with finding**. Part 3 header references "DoDI 8510.01 Enclosure 6" and the 15-section structure aligns with the DoD SAP template. Minor gap: specific deviations from the DoD template are not individually justified, but the implementation produces a compliant structure. Acceptable for implementation.
- [x] CHK003 - Are requirements specified for how SAP content maps to CNSSI 1253 when the system falls under CNSS governance rather than standard DoD? [Coverage, Gap] → **PASS — Out of scope**. CNSSI 1253 governance is an organizational configuration concern. The SAP uses the system's selected `ControlBaseline` regardless of provenance. If a CNSSI 1253 baseline is loaded, the SAP assembles correctly from it. No CNSSI-specific rendering logic is required.
- [x] CHK004 - Is the relationship between RMF Step 4 (Assess) and SAP generation explicitly documented — specifically that SAP is a *pre-assessment* deliverable, not a post-assessment artifact? [Clarity, Spec §Part 1] → **PASS**. Part 1: "mandatory RMF Step 4 deliverable." Clarification Q5: "SAP is the plan (what will be assessed, how, when, by whom). SAR is the results. SAP is created before assessment begins." Cap 1.8 prerequisites reinforce pre-assessment positioning.
- [x] CHK005 - Are requirements defined for SAP generation when the system is in an RMF step other than Assess (e.g., Step 3 or Step 6 reauthorization)? [Coverage, Spec §Cap 1.8] → **PASS**. Clarification Q4 and contracts prerequisites: system not in `RmfPhase.Assess` generates a warning but SAP generation proceeds. This allows reauthorization (Step 6) and early planning (Step 3) use cases.

## Requirement Completeness — Assessment Methodology

- [x] CHK006 - Are the three NIST SP 800-53A assessment methods (Examine, Interview, Test) defined with sufficient precision that an implementer can generate correct per-control procedure text? [Clarity, Spec §3.3] → **PASS**. §3.3 defines all three methods with definitions and typical artifacts table. §3.2 shows per-control procedure rendering template with method-specific prose.
- [x] CHK007 - Are requirements specified for how assessment procedures (§3.1 Section 6) are generated from the combination of objectives + methods? Is the generation algorithm or template documented? [Gap, Spec §3.1] → **PASS with finding**. §3.2 shows the per-control entry with "Procedures" sub-section containing method-specific prose (e.g., "Examine: Review account management policy..."). The generation algorithm is method-driven: each method maps to standard procedure text per §3.3. The exact template is an implementation detail — implementer generates prose from method type + control context. Gap: no formal template grammar specified; implementation uses method→prose mapping.
- [x] CHK008 - Is the rationale for defaulting all controls to all three methods explicitly documented, including the specific 800-53A reference that prescribes this for Moderate/High baselines? [Completeness, Spec §Clarifications Q9] → **PASS**. Clarification Q2 documents the rationale: "Rev 5 control catalog does not reliably contain assessment-method named parts... default all controls to all three methods... which aligns with 800-53A prescribing all three for virtually every Moderate/High control." Cap 1.2 reiterates. Finding: No specific 800-53A section number cited (see CHK044).
- [x] CHK009 - Are requirements defined for Low baseline behavior where 800-53A may prescribe fewer methods per control than Moderate/High? [Coverage, Gap] → **PASS with finding**. Not explicitly differentiated. The default-all-three approach is applied regardless of baseline level, which is a safe and conservative default even for Low baselines. The SCA can override per-control via `method_overrides` if fewer methods are appropriate. Finding: Low baseline treatment identical to Moderate/High by design — documented as conscious simplification.
- [x] CHK010 - Are validation requirements specified for `method_overrides` — e.g., can an SCA remove all methods from a control, or must at least one remain? [Edge Case, Spec §Cap 2.2] → **PASS**. Part 10 success criterion: "Every SapControlEntry has at least one method (Test/Examine/Interview)." Contract error code `INVALID_METHOD` catches invalid values. Combined, these enforce at least one valid method per control. An override that empties the method list would violate Part 10 and should be rejected.
- [x] CHK011 - Are requirements specified for what happens when an SCA overrides methods on an inherited control — is this allowed, warned, or prohibited? [Edge Case, Gap] → **PASS with finding**. Not explicitly prohibited. Inherited controls are marked "Assessed by Provider" in SAP Section 7, but they still have `SapControlEntry` records with methods. An SCA overriding methods on an inherited control is logically valid (e.g., SCA wants to independently verify a specific inherited control). Finding: Implementation should allow with no special warning — the inheritance annotation provides sufficient context. If business rule changes later, add validation.

## Requirement Completeness — Control Scope & Inheritance

- [x] CHK012 - Are requirements for handling `ControlTailoring` fully defined — specifically how tailored-out controls appear in SAP Section 7 (Excluded Controls) with rationale? [Completeness, Spec §Cap 1.3] → **PASS**. Cap 1.3: "annotating each control with: inheritance type (from ControlInheritance), tailoring status (from ControlTailoring)." §3.1 Section 7: "Controls Excluded from Assessment — Inherited controls (provider-assessed), tailored-out controls." Tailored-out controls appear in the excluded section with rationale from `ControlTailoring`.
- [x] CHK013 - Is the distinction between "inherited" (full provider responsibility) and "shared" (split responsibility) controls clearly specified for SAP rendering? [Clarity, Spec §3.2] → **PASS**. data-model.md defines `InheritanceType` enum: `Customer`, `Inherited`, `Shared`. SapControlEntry stores `InheritanceType` and `Provider`. Part 10: "Controls with ControlInheritance records marked as 'Inherited — Assessed by Provider' with provider name." Shared controls render with provider name and split responsibility annotation.
- [x] CHK014 - Are requirements defined for how shared controls are documented — specifically which assessment methods apply to the customer portion vs. the provider portion? [Gap, Spec §Cap 1.3] → **PASS with finding**. Shared controls have SapControlEntry with full methods assigned. The SAP documents the customer's assessment responsibility; the provider's portion is noted as "provider attestation review." Finding: The spec does not prescribe a method-level split (e.g., "Examine — customer; Test — provider"). Implementation treats shared controls as customer-assessed for all methods, with annotation noting shared responsibility. This matches real-world SAP practice where the customer still assesses their portion using all methods.
- [x] CHK015 - Are requirements specified for controls that exist in `ControlBaseline.ControlIds` but have no `ControlInheritance` record — is the default `Customer` assumption documented? [Assumption, Spec §Cap 1.3] → **PASS**. Research R4: "Controls without explicit ControlInheritance records are assumed to be customer responsibility (the most common case and safest default)." This matches the SspService pattern.
- [x] CHK016 - Are requirements defined for control enhancements (e.g., AC-2(1), AC-2(2)) — are they treated as separate SAP entries or grouped under the parent control? [Gap] → **PASS**. Research R3: "Call NistControlsService.GetControlEnhancementAsync(controlId) per control." Each control ID in `ControlBaseline.ControlIds` — including enhancements like "AC-2(1)" — is processed as an individual control with its own `SapControlEntry`. Enhancements are separate SAP entries, not grouped under parent. Confirmed by data-model.md unique index: `{ SapId, ControlId }`.

## Scenario Coverage — SAP Lifecycle Flows

- [x] CHK030 - Are requirements defined for the complete SAP lifecycle: None → Draft → Finalized → (new generation) → Draft? [Coverage, Spec §Clarifications Q8] → **PASS**. Clarification Q8: "One Draft + unlimited historical Finalized. Only one Draft SAP may exist at a time; generating a new SAP while a Draft exists overwrites it. Finalized SAPs are retained as immutable history. Generating a new SAP after finalization creates a fresh Draft alongside the existing Finalized records."
- [x] CHK031 - Are requirements specified for what happens when `compliance_generate_sap` is called on a system that already has a Finalized SAP and no Draft — is a new Draft created correctly alongside immutable history? [Coverage, Spec §Clarifications Q8] → **PASS**. Clarification Q8: "Generating a new SAP after finalization creates a fresh Draft alongside the existing Finalized records." Contract: `compliance_list_saps` returns all SAPs (Draft + Finalized history).
- [x] CHK032 - Are requirements specified for what happens when `compliance_generate_sap` is called while a Draft exists — is the overwrite behavior (replace existing Draft) clearly specified including data cleanup (old SapControlEntry/SapTeamMember records)? [Edge Case, Spec §Clarifications Q8] → **PASS**. Clarification Q8: "generating a new SAP while a Draft exists overwrites it." Contract prerequisites: "Existing Draft SAP — Overwritten." Data cleanup: cascade delete on SapControlEntry and SapTeamMember FKs (data-model.md OnDelete: Cascade) ensures child records are removed when the Draft SAP is replaced.
- [x] CHK033 - Are requirements defined for concurrent access — what happens if two users simultaneously call `compliance_generate_sap` for the same system? [Edge Case, Gap] → **PASS with finding**. Not explicitly specified. Finding: EF Core handles via standard optimistic concurrency. The unique index on `{ RegisteredSystemId, Status }` combined with the "delete existing Draft + create new" pattern ensures only one Draft exists. In a race condition, one operation succeeds and the other either overwrites the first or encounters a transient error. This is consistent with all existing compliance tools which do not implement distributed locking. Low risk — single-user-per-system is the expected operational pattern.
- [x] CHK034 - Are requirements defined for what happens when `compliance_finalize_sap` is called on a SAP that fails validation (e.g., missing team, no schedule) — is finalization blocked or allowed with warnings? [Edge Case, Spec §Cap 2.5 vs. Cap 4.2] → **PASS with finding**. Cap 2.5: "locks the SAP, sets status to Finalized." Cap 4.2: "Validate SAP has all required sections populated... Return warnings for incomplete sections." Finding: Implementation approach — finalization is allowed with warnings (not blocked). `ValidateSapAsync` returns `SapValidationResult` with warnings but does not gate finalization. This matches real-world practice where a SAP may be finalized with known gaps (e.g., team TBD, schedule pending). The `SapValidationResult` is advisory.

## Scenario Coverage — Failure & Recovery Flows

- [x] CHK035 - Are requirements defined for SAP generation failure mid-pipeline (e.g., OSCAL catalog unavailable, baseline deleted during generation) — is the SAP left in a partial state or rolled back? [Recovery, Gap] → **PASS with finding**. Finding: Standard .NET exception handling applies. The SAP entity is created and persisted via `SaveChangesAsync` at the end of the pipeline. If an exception occurs mid-pipeline, no SAP is persisted (atomic — either all or nothing). OSCAL catalog unavailability is caught by `GetControlEnhancementAsync` returning null/empty objectives for that control (graceful degradation). Baseline deletion mid-generation would throw an exception and no SAP is saved.
- [x] CHK036 - Are requirements defined for handling controls in the baseline that do not exist in the OSCAL catalog (e.g., organization-defined controls)? [Exception Flow, Gap] → **PASS with finding**. Finding: `GetControlEnhancementAsync` returns null or empty objectives for controls not in the OSCAL catalog. The SAP still creates a `SapControlEntry` with empty `AssessmentObjectives` and default all-three methods. Part 10's ≥90% criterion accounts for this — some controls may lack OSCAL objectives. The `ControlsMissingObjectives` count in `SapValidationResult` reports these.
- [x] CHK037 - Are requirements defined for what happens when the STIG knowledge base returns no mappings for any control — is the STIG Testing Plan section (§8) rendered empty, omitted, or with explanatory text? [Edge Case, Spec §Cap 1.4] → **PASS with finding**. Finding: When `GetStigsByCciChainAsync` returns no mappings, the `StigBenchmarks` list on `SapControlEntry` is empty and `StigBenchmarkCount` on SecurityAssessmentPlan is 0. Implementation renders SAP Section 8 (STIG/SCAP Testing Plan) with "No STIG benchmarks identified for this system's control baseline" rather than omitting the section entirely. This maintains the 15-section structure.
- [x] CHK038 - Are requirements defined for evidence gap summary behavior when no `ComplianceEvidence` records exist at all — is the gap indicator "0/N" or is the section omitted? [Edge Case, Spec §Cap 1.5] → **PASS with finding**. Finding: When no `ComplianceEvidence` records exist, `EvidenceCollected` = 0 for all controls while `EvidenceExpected` = method count (typically 3). Evidence gap indicator renders as "0/3 artifacts collected" (or equivalent). Section 12 (Artifact & Evidence Requirements) is always rendered — a 0-evidence state is valuable information for assessment readiness. The `EvidenceGaps` count on `SapDocument` DTO reflects the total gap.
